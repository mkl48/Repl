01 ReplClient

Repl — the umbrella. require(ReplicatedStorage.Repl) returns the whole public API. It's realm-aware: the shared engine (Registry, Dispatcher, Types, Theme) loads everywhere, and the Fusion Bar is pulled in only on the client, only when you call Repl:Start() — so the server never touches Fusion.

example
local Repl = require(game.ReplicatedStorage.Repl)
Repl.Theme:Set("tokyonight")
Repl:RegisterCommand({ Name = "hello", Args = { "who: String? = world" },
    Run = function(ctx, who) return ctx:ok(`hello, {who}!`) end })
Repl:Start()                       -- mounts the bar (client); no-op on server

Dev-facing state + events (Repl.IsFocused(), Repl.FocusChanged, …) let a game gate its own keybinds while the terminal is in use.

RETURNS boolean

Registers a custom argument type.

Adds a "BeforeRun" / "AfterRun" hook.

function Repl:Execute(line: string, exec: any?)Client#

Runs a line programmatically (same path the bar submits through).

02 AstClient

AST node kinds for the Repl shell. The Parser builds plain tables tagged with a kind from Ast.Kind; the Dispatcher walks it.

03 LexerClient

The Repl shell lexer: source text → a flat list of neutral Tokens.

Neutral means no FLAG / DIRECTIVE / SLASH_REF / SLASH_DIV tokens: -, --, /, // come out as plain operator tokens and the Parser assigns meaning by context and by spaceBefore adjacency. The lexer never throws — malformed input yields diagnostics and the scan continues, which also backs analysis.

function Lexer.tokenize(source: string): LexResultClient#

Tokenizes source. Always returns tokens (ending in an EOF token) plus any diagnostics; it never errors.

RETURNS LexResult

04 ParserClient

Recursive-descent parser for the Repl shell: Tokens → Ast + diagnostics.

The parser is where context lives: the same / token becomes a reference in value position and division in operator position; && is chain control between statements and boolean inside an expression. Nothing here throws — errors are collected as diagnostics with best-effort recovery, so this also powers analysis/highlighting.

function Parser.parseTokens(tokens: { Token.Token }, diagnostics: { Lexer.Diagnostic }): ParseResultClient#

Parses a token stream into an Ast, collecting diagnostics (never throws).

RETURNS ParseResult

function Parser.parse(source: string): ParseResultClient#

Tokenizes and parses source in one call.

RETURNS ParseResult

05 ReturnPoolClient

The return pool. Commands push typed/labelled return records; / references read them back. There is no positional stack — you reference returns by what they are (// last, /Player by type, /target by label, #n for the nth-most-recent, .path to drill in). Failed links contribute nothing, so the pool only holds real results.

function ReturnPool.new()Client#

Creates an empty pool.

function ReturnPool:clear()Client#

Empties the pool (once per top-level line).

function ReturnPool:push(value: any, valueType: string?, label: string?, fromCommand: string?)Client#

Appends a successful return. A nil value is still a valid result and recorded.

function ReturnPool:last(): Record?Client#

The most recent record, or nil.

RETURNS Record?

function ReturnPool:resolveRecord(ref: Reference): Record?Client#

Resolves a reference to its record (a label takes precedence over a type on a name clash).

RETURNS Record?

function ReturnPool:resolve(ref: Reference): (any, string?)Client#

Resolves a reference to a value, applying any .path. Returns (value, nil) on success or (nil, errorMessage) on a missing return or a bad access.

RETURNS (any, string?)

06 TokenClient

Token kinds and the Token shape for the Repl shell lexer.

The lexer is deliberately neutral: it never decides whether a token means "reference" vs "division" or "flag" vs "minus". It emits the smallest tokens plus spaceBefore adjacency, and the Parser assigns meaning by context and token position.

type TokenClient#

One lexed token. value is the cooked value for literals; spaceBefore records whether whitespace separated this token from the previous one (so the parser can tell -s the flag from - s the operator).

definition
export type Token = {
	kind: Kind,
	text: string, -- raw source text matched
	value: any, -- cooked value: number | string | boolean | nil
	pos: number, -- 1-based byte offset of the first char
	line: number,
	col: number, -- 1-based byte column within the line
	spaceBefore: boolean, -- was this token separated from the previous by whitespace?
}

07 ShellClient

The shell front end: source → tokens → AST. Re-exports the stages (Lexer, Parser, Ast, Token) and the two entry points.

08 ExprClient

Parses a type expression — the text after the colon in an Args entry — into a small AST the Types engine compiles. This is what makes Player | Character, Number & Int32, and 2 < Number < 30 real.

Grammar (low → high precedence): Union(|) → Intersection(&) → Range(bounded atom) → Atom(Name | "(" expr ")"). A trailing ? on the whole expression marks the argument optional.

function Expr.parse(source: string): ParseResultClient#

Parses a type-expression string. Never throws; returns the AST (or nil), whether the argument is optional (trailing ?), and any errors.

RETURNS ParseResult

09 RobloxClient

Built-in Roblox argument types, registered onto the Types engine at startup so Args = { "victim: Player" } works out of the box — with autofill.

A type has three jobs and autocomplete is what fills in suggestions: it takes the partial text and returns the matching options. Player lists online player names, Players adds all/*, Team lists team names. transform turns the chosen text into the real instance, and validate confirms a value is the right kind (both must actually return — a validate that returns nothing reads as "invalid").

function Roblox.register(Types: any)Client#

Registers Player / Players / Team on a Types engine.

10 TypesClient

The type engine. A type is a small record with three jobs: transform turns raw text into a value, validate checks a value, and autocomplete offers completions for a partial input. The engine ships primitive types and compiles a type expression (Player | Character, Number & Int32, 2 < Integer < 30) into a single composed type — unions merge completions, intersections stack constraints, and a bounded range enumerates its valid values so the command bar can suggest them.

type TypeClient#

A resolved, ready-to-use type.

definition
export type Type = {
	name: string,
	optional: boolean?,
	--- Raw text → `(value, nil)` or `(nil, error)`.
	transform: (raw: string) -> (any, string?),
	--- A value → `(ok, reason?)`.
	validate: (value: any) -> (boolean, string?),
	--- A partial input → completion strings.
	autocomplete: (prefix: string) -> { string },
}
type DefClient#

A type definition you register. Only name and transform are required; the rest have sensible defaults.

definition
export type Def = {
	name: string,
	transform: ((raw: string) -> (any, string?))?,
	validate: ((value: any) -> (boolean, string?))?,
	autocomplete: ((prefix: string) -> { string })?,
	--- Marks a whole-number type so ranges over it enumerate integers.
	integer: boolean?,
}
function Types.register(def: Def)Client#

Registers (or replaces) a type definition.

function Types.get(name: string): Def?Client#

Returns the raw definition registered under name, if any.

RETURNS Def?

function Types.compile(source: string): Type & { parseErrors: { string } }Client#

Compiles a type-expression string into a resolved Type. On a parse error the returned type rejects everything, and the messages are in .parseErrors.

RETURNS Type & { parseErrors: { string } }

11 ArgumentClient

Parses the terse Args DSL — one "name: TypeExpr = default" string per argument — into structured specs. The text after the colon is a real type expression (so amount: Integer? = 1 and n: 2 < Integer < 30 both work); the type is compiled here so the handler receives typed, validated values. Never throws; problems land in each spec's .error.

function Argument.parse(spec: string): SpecClient#

Parses a single entry: a positional "name: TypeExpr = default # desc" or a flag "-x, --name: TypeExpr = default # desc" (a bare --name is a boolean).

RETURNS Spec

function Argument.usage(spec: Spec): stringClient#

A usage token for one spec: <name> (required) or [name] (optional).

RETURNS string

function Argument.parseList(specs: { string }): ({ Spec }, { string })Client#

Parses a whole Args list, collecting per-entry errors plus one structural rule: a required argument may not follow an optional one.

RETURNS ({ Spec }, { string })

12 CommandClient

Validates and normalizes a command definition — the table a user returns from a command module — into a checked record. Names are validated, the Args DSL is parsed, and every problem is collected (never thrown) so the Registry can report a whole module's mistakes at once.

13 RegistryClient

The Registry holds every registered command, indexed by name and by alias, and is the one place a name clash is caught. It parses each definition through the Command validator, exposes custom-type registration, and keeps the before/after run hooks the dispatcher fires. Registration never throws: it returns the collected errors so a bad module fails loudly but in isolation.

RETURNS Command.Command?

14 ContextClient

The Context (ctx) handed to every command handler: it is how a command talks back to the terminal. Output methods emit structured Output nodes; ok/err/abort both report and return a Result with the right exit code; returns records a value in the ReturnPool for later /references; store is a per-session scratch space. Interactive calls (confirm, prompt, choose, exec) delegate to an injected bridge — headless in tests, a real client round-trip once Network is in.

function Context.new(config: Config?)Client#

Creates a context. Everything is optional; a fresh ReturnPool and an empty store are provided if you don't pass them.

function Context:emit(node: Output.Node): Output.NodeClient#

Emits a node: records it and forwards it to the sink (the live terminal).

RETURNS Output.Node

function Context:reply(text: string)Client#

A neutral reply line.

function Context:custom(color: string, text: string)Client#

A line in an arbitrary theme color token.

function Context:icon(name: string, text: string)Client#

function Context:table(rows: { { string } })Client#

function Context:ascii(text: string)Client#

A multi-line ascii block (split on newlines).

RETURNS Result.Result

Runs another command by name from inside this one.

RETURNS any

function Context:subscribe(event: string, handler: (...any) -> any)Client#

Listens for event. Handlers run for same-realm posts and posts from the other side (routed in by the network layer); any earlier post is replayed, so subscribing late never misses one. A handler's return value answers a post.

function Context:post(event: string, ...: any)Client#

Posts event to both sides. Returns an awaitable: :Await() blocks for a guaranteed value — the other realm's response over a RemoteFunction when a shared command spans realms, or the local subscriber's response otherwise — so there's no race between posting and a listener being ready.

15 DispatcherClient

The Dispatcher runs a submitted line. It tokenizes with the Lexer, splits on chain operators, and for each command: resolves it in the Registry, binds the argument tokens (coercing each through the arg's compiled type, filling defaults, erroring on a missing required one), runs the handler with a Context, and captures a Result. Chains branch on the exit code — && runs the next only on success, || only on failure, ;/& always. A shared return pool lets // / /refs read earlier results back. Fully headless-testable.

Pipelines (|) and rich reference paths (/x.y#2) are not wired yet — bare // and /Name references resolve; anything fancier is left for later.

function Dispatcher.new(registry: any, config: Config?)Client#

Creates a dispatcher over registry.

function Dispatcher:run(line: string, exec: ExecConfig?): anyClient#

Runs a whole line (with chains). Returns the last executed Result.

RETURNS any

16 BarClient

The command bar — the visible surface. A bottom-anchored terminal: a scrolling output area, a prompt + TextBox input with a custom block cursor and inline ghost text, and an autocomplete popover with a highlighted, arrow-cyclable selection. While the bar is focused it takes over input — character movement, shift-lock, and the backpack hotkey are suppressed so a keystroke never leaks to the game. Enter submits to the injected submit callback; Ctrl dismisses the suggestions; Backquote toggles; Escape hides. Output is drawn by the Terminal renderer and recolors live with the Theme. Exposes dev-facing state (Focused, Visible) and events (Opened, Closed, FocusChanged, Submitted). Studio-only (pulls Fusion).

17 CompleterClient

The completion engine behind the bar. Given the current input, it decides whether the cursor is on the command name (fuzzy-matched against the Registry) or on an argument (whose compiled type supplies the suggestions — including a range's enumerated values), and returns ranked completions with the text span each one replaces. Pure logic: the bar draws the results, this decides them, so it's fully headless-testable.

type CompletionClient#

One suggestion. replaceStart/replaceEnd are 1-based byte offsets into the input that the bar overwrites when the completion is accepted.

definition
export type Completion = {
	text: string,
	kind: "command" | "argument",
	detail: string, -- a usage/type hint shown beside the entry
	description: string, -- a longer description, for the intellisense line
	replaceStart: number,
	replaceEnd: number,
}
type HintClient#

Signature intellisense for the command currently being typed.

definition
export type Hint = {
	name: string,
	signature: string, -- "name <a> [b]"
	description: string,
	args: { any }, -- the command's arg specs (name/typeExpr/optional/description)
	activeIndex: number, -- which arg the cursor is on (0 = still on the name)
}
function Completer.complete(registry: RegistryLike, input: string): { Completion }Client#

Produces completions for input against registry (cursor assumed at end).

RETURNS { Completion }

function Completer.hint(registry: RegistryLike, input: string): Hint?Client#

Signature intellisense for input: the resolved command, its usage, and which argument the cursor is on (0 while still typing the name). Returns nil when the first word isn't a known command.

RETURNS Hint?

function Completer.ghost(registry: RegistryLike, input: string): stringClient#

The ghost-text completion of input: the remaining characters of the single best suggestion, if it extends the current word (case-insensitive prefix). Returns "" when there's nothing to ghost.

RETURNS string

18 HistoryClient

Command history for the bar — readline semantics. add records a submitted line (consecutive duplicates and blanks dropped, capped to a limit); up/down walk older→newer, and the in-progress line is saved as a draft so pressing Down past the newest entry restores what you were typing. Pure state, headless-testable.

function History.new(limit: number?)Client#

Creates a history buffer (default cap 100 entries).

function History:add(text: string)Client#

Records a submitted line and resets navigation. Blank lines and immediate repeats are ignored.

function History:up(current: string): string?Client#

Moves to an older entry, stashing current as the draft on the first step. Returns the entry to show, or nil when there's no history.

RETURNS string?

function History:down(): string?Client#

Moves to a newer entry; stepping past the newest restores the draft (and returns to the live line). Returns nil when not currently navigating.

RETURNS string?

function History:reset()Client#

Returns to the live line without changing the stored entries.

function History:all(): { string }Client#

Every entry, oldest first (a copy).

RETURNS { string }

19 ClientClient

The client realm: the visible command Bar plus the headless pieces that drive it — the Completer (autocomplete) and History. The realm-aware Repl.Start wires a Bar to the dispatcher and network once those land; until then, mount a Bar directly with Client.Bar.new{...}.

Requiring this pulls Fusion (via Bar), so it's client/Studio-only — the server realm never touches it.

20 IconsClient

The icon map: icon name → Roblox asset id. The PNGs live in assets/icons/png (white, so they tint to the theme via ImageColor3) — upload them and fill in the ids below, or set them at runtime with Icons.set / Repl.Icons.set(name, "rbxassetid://…"). Until an id is set, the Terminal falls back to rendering [name] as text, so nothing breaks.

Semantic aliases (success, error, warn, prompt, …) map onto the base glyphs so ctx:icon("success", …) and Output line kinds resolve without knowing the underlying icon.

Resolves an icon name (or alias) to its asset id, or nil if unset.

RETURNS string?

function Icons.set(name: string, assetId: string | number)Client#

function Icons.setMany(map: { [string]: string | number })Client#

Bulk-set from a { [name] = id } table.

21 OutputClient

The structured output model. Commands never print strings — they emit Segments (styled text runs), grouped into Lines, optionally grouped into a Block (a table, an ascii art, a status panel). Colors are semantic tokens ("success", "error", "accent", "muted"…), not Color3, so the same output recolors live when the theme changes and stays fully headless. toText flattens any node to plain text for logs and no-UI fallbacks.

22 TerminalClient

The terminal renderer: maps the headless Output tree (Segment/Line/ Block) to Fusion instances. A Line becomes one RichText label whose text is a Computed — so bold/italic/underline render inline and every segment's Theme color re-evaluates the moment the scheme changes. A Block becomes a vertically-stacked frame of lines. Studio-only (pulls Fusion).

Icons (Lucide) are not yet drawn as images — a segment's icon is rendered as a [name] text marker until the Lucide asset map lands.

24 RenderClient

25 ThemeClient

The Theme. Every color in Repl — terminal output, the bar chrome, every Output Segment, every Repl.UI widget — is named by a semantic token (accent, success, error, bg, prompt…), never a literal color. A theme maps those tokens to a palette; Set swaps the palette and notifies listeners, which is what makes the whole surface recolor live. Ships tokyonight, gruvbox, catppuccin, and nord.

The palette values are hex strings so the engine stays headless; Color3 resolves them at the Roblox boundary (Theme:Color3(token)).

29 EventBusClient

A queued event bus — the core of shared-command communication. Handlers subscribe to named events; post delivers to everyone listening and retains the payload, so a handler that subscribes late still receives every earlier post (no race between a fast post and a slow subscribe). Each subscriber sees each post exactly once, in order.

This is the realm-neutral core; the shared-command wrapper layers a RemoteFunction round-trip on top so a cross-realm post returns a guaranteed value. Headless-testable on its own.

30 FuzzyClient

The fuzzy matcher behind autocomplete. A query matches a candidate when its characters appear in order (a subsequence); the score rewards matches at the start, on word boundaries (after . _ - / space, or a camelCase hump), and in consecutive runs — so pk ranks players.kill above a stray mid-word hit.
